## Future work (not on the numbered build order)

Genuinely future-facing directions, deliberately not treated as numbered phases — no committed design, scope, or sequencing yet. See `Backlog.md` for smaller, more concretely-scoped deferred items instead.

- **Front-end data visualization module** — deprioritized (2026-07-14), not built. Users would connect to a serve-layer view/table, pick a chart type/axes/legend, and the app renders via Plotly Express. Open decisions, not yet made: connectivity (ADBC vs. the plain `trino` client — the Trino ADBC driver isn't pip-installable, needs a separate binary install step, in tension with this project's all-`uv` tooling) and chart-config persistence (ephemeral per-session vs. saveable, which would need a new metadata table).
- **Unified security model, Azure Entra-integrable** — one security model defined once and applied platform-wide (frontend, metadata Postgres, Trino/Polaris, Kubernetes), rather than each subsystem managing its own access control independently. Proposed shape: the frontend centralizes role assignment, sourced from consumer user identities in Azure Entra; that role-assignment management generates role definitions, which in turn generate RBAC roles, security groups, and group assignments via infrastructure as code (see the IaC item below). Not scoped — open questions include how Entra identities map into this platform's own metadata schema, whether role definitions are platform-wide or per-module, how changes propagate, and how this interacts with Trino/Polaris's own native authorization versus Kubernetes RBAC versus the frontend's own access control.
- **Infrastructure as Code** — Terraform vs. Bicep not yet decided. Today's infrastructure (kind cluster, Kubernetes manifests, Helm installs) is entirely bash/kubectl/helm/`just`-recipe-driven, not IaC-managed. A real Azure deployment (the portability goal noted above) would need this; it's also the natural implementation vehicle for the security model item above (RBAC roles/security groups/group assignments as code).
- **Spark as a full alternative engine, staging/model layers included** — today, `data_feed.processing_engine` already lets a feed opt into Spark instead of Polars, but only for the raw→clean transform (Phase 6 above). This would extend that choice into staging/model, where dbt is currently the only option (`dbt-trino` has no Python model execution backend — a day-one architectural decision, not a gap) — Spark and Polars both becoming viable alternatives to hand-written dbt SQL for staging/modeling logic. A bigger structural change than the existing raw/clean opt-in: a new code path parallel to dbt entirely, not an extra dbt materialization option. Related to `Backlog.md`'s "Non-dbt staging engine (PyIceberg) for complex staging logic" entry, which already investigated a narrower version of this same problem (PyIceberg upsert only, not general Spark/Polars staging) — worth reading together, not duplicating.
- **Hierarchy-tiered, dependency-driven `master_pipeline` execution** — `data_feed.batch_feed_hierarchy` and `lakehouse_models.batch_hierarchy` are currently inert metadata; nothing in Dagster reads either column. Today, a `batch_group`-triggered `master_pipeline` run is fully sequential and globally barriered: every feed extracts one at a time, *then* every domain's entire staging+model dbt build runs as one job, *then* serving. Proposed redesign, scoped to `orchestration_kind="batch_group"` runs that feed real hand-modeled `lakehouse_models` rows (ODS-producing batch groups and `model_schema`-triggered runs keep today's behavior unchanged — this mechanism doesn't apply to either):
  - **Feeds**: tier by `batch_feed_hierarchy`. Tier N feeds extract in parallel; tier N+1 only starts once every tier-N feed reaches a terminal status (a failure in tier N blocks tier N+1, since tier-to-tier order is a blanket assertion with no finer-grained signal — but doesn't cancel already-running tier-N siblings).
  - **Models**: each `lakehouse_models` row already has real per-model feed dependencies via `depends_on_feeds` — after *each* feed tier completes (not after the whole extraction phase), check every not-yet-launched model; if its `depends_on_feeds` are now all satisfied, launch its own staging+model build immediately, concurrently with later feed tiers still extracting feeds needed by *other* models. A model whose feeds are all early-tier never waits on unrelated late-tier feeds. Among ready models: dims before facts (`table_type`, a baseline rule), then `batch_hierarchy` tier within each `table_type`, same tier-blocks-next-tier/siblings-independent rule as feeds.
  - **Failure semantics**: a failure only blocks what actually depends on it — unrelated branches run to completion; `master_pipeline`'s own final status still reports failure if anything failed anywhere, without cutting short unrelated in-flight work.
  - **Architecture**: one interleaved loop inside `master_pipeline`'s existing single op — not a separate "child pipeline" job per model. A feed shared by multiple models still only gets extracted once per run for free (there's exactly one global launch per feed), so no cross-run coordination/dedup mechanism is needed. Needs: a new parallel launch/wait helper (today's `launch_and_wait` only submits and polls one job at a time), and new per-model `MODEL_BUILD_JOBS`/`MODEL_SERVE_JOBS` (`AssetSelection.assets(AssetKey(model)).upstream()`) added alongside — not replacing — today's whole-domain `MODELING_JOBS`/`SERVING_JOBS`.
  - **Operational gaps to resolve as part of this work**: nothing today caps *total* concurrent pods, only same-feed concurrency (the per-feed pool's `default_limit: 1`) — this design increases how many independent pods can be in flight at once, and needs a real ceiling. `dagster-webserver`/`dagster-daemon` are both hard-capped at 1 replica (`dagster-daemon` isn't KEDA-managed at all) and become a real bottleneck for many concurrent job launches/status polls at this fan-out. Per-pod startup overhead (scheduling, image pull, container/interpreter init) versus small actual model-build work size is a real tradeoff of this fine-grained approach, not free parallelism. No connection-limit tuning exists today for Postgres/Trino/Polaris/MinIO against a higher concurrent-pod count. This project's local `kind` cluster is also a hard, fixed-size single-node ceiling (one Docker container, fixed CPU/memory) — genuinely cannot exercise "hundreds of concurrent pods" at all locally; real load behavior can only be validated against a real multi-node deployment, tying this directly to the Infrastructure as Code / node-autoscaling item above.
  - Also surfaced while researching this, worth fixing alongside: `Learnings.md`'s "two feed tags" entry references a nonexistent `AssetSelection.groups(feed).upstream()` call.
