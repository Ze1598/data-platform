# Walkthrough: Onboarding a New Streaming Source

A complete, copy-paste-able walkthrough of the platform's real-time
ingestion path — from registering a Kafka topic as metadata, through
schema discovery and deploying a real Flink sink, to writing the
hand-authored serve-layer join that makes the stream actually useful. The
worked example is `inventory_events`: a real-time feed of inventory
quantity changes per branch, joined against the already-persisted
`sales_dim_branch` model table for city enrichment — the same join shape
`sales_events` (the platform's original streaming proof) already uses.

This source already exists on any platform that's been through a normal
`just start` (it's seeded automatically by `scripts/seed_metadata_db.py`,
alongside `sales_events`) — so the values below describe both **how it's
actually configured today** and **exactly what to enter to reproduce it
from a fresh, unseeded platform**. If you follow the UI steps below
against an already-seeded platform, the **Add new** insert will fail on a
duplicate `friendly_name` — that's expected, and means you can skip
straight to [step 5](#5-deploy-the-ingestion-plumbing-command).

This exercise is worth doing specifically for one thing: **the same
"metadata config + business logic, boilerplate generated" story a batch
feed gets (see `Walkthrough_Metadata_Ingestion.md`), applied to a
continuous stream instead of a bounded run.** Everything up to the serve
view (topic registration, Flink SQL sink, `FlinkDeployment` CR, Kafka
topic creation) is generated by `scripts/generate_streaming_ingestion.py`
from metadata alone — zero hand-written plumbing. The only thing you write
by hand is the serve view's join, exactly like a batch fact's join to its
dimension.

**Streaming is a hot path, not a replacement for batch.** `streaming_source`
is a deliberately separate concept from `data_feed` — see
`metadata/DataModel.md`'s `streaming_source` section for why (its
`extraction_type`/watermark-based columns all assume a discrete, bounded
run, which a continuous stream isn't). If you also want this data folded
into a `model` table as one of several sources (a nightly catch-up, or a
periodic/sensor-driven batch job absorbing streaming payloads), that's a
new batch source reading from `streaming.*` like any other source — not
covered by this walkthrough, which is scoped to the streaming path itself.

Steps marked **[UI]** are done in the Streamlit frontend. Steps marked
**[Code]** require writing a file by hand — same split as
`Walkthrough_Metadata_Ingestion.md`: this platform's ingestion-plumbing
codegen is metadata-driven, but the business-logic SQL inside a serve view
is not. Steps marked **[Command]** are run from the repo root,
`/Users/josecosta/Documents/projects/data-platform/`.

## 0. Bring up the platform

```bash
just start
```

`streaming::start` (part of `just start`'s default sequence) brings up
Kafka and the Flink Kubernetes Operator unconditionally, but — unlike
batch feeds — **never deploys a real `FlinkDeployment` on its own**, even
for a source that ran successfully before a nuke. `schema_registry` is
deliberately never seeded for a `streaming_source` row, same rule as
`data_feed` ("discovery bootstraps it, no hand-written baseline needed") —
so every source needs schema discovery (steps 3–4 below) run by hand
before `scripts/generate_streaming_ingestion.py` will consider it ready to
deploy. See `Backlog.md` if this becomes a recurring friction point worth
automating further for real (non-test) usage.

### Accessing running services

Same table as `Walkthrough_Metadata_Ingestion.md` — every service is
reachable at `localhost:<port>` with no `kubectl port-forward` needed. This
walkthrough additionally needs:

| Service | Address | Notes |
|---|---|---|
| Streamlit (frontend CRUD) | `http://localhost:8501` | **Streaming Sources** page |
| Kafka | in-cluster only, `kafka.streaming.svc.cluster.local:9092` | No NodePort — the frontend's own "Discover schema" action and this walkthrough's manual message production both go through `kubectl exec` into the Kafka pod, not a host-side client |

## 1. Register the streaming source **[UI]**

Streamlit (`http://localhost:8501`) → **Streaming Sources** → *Add new*.

| Field | Value |
|---|---|
| Friendly name | `inventory_events` |
| Kafka topic name | `inventory-events` |
| Table name | `inventory_events` |
| Model schema (domain) | `sales` |
| Event timestamp column | *(not shown yet — appears only after schema discovery has run, step 4)* |
| JobManager memory / TaskManager memory / TaskManager CPU / Parallelism | *(leave all blank — falls back to the platform default, `1024m`/`1024m`/`0.5`/`1`)* |
| Autoscaler enabled | leave unchecked — Flink's own built-in autoscaler, flagged experimental in the operator's current docs, opt-in only |
| Active | **on** |

No credentials or connection config to provide — this platform has exactly
one shared Kafka broker (`kafka.streaming.svc.cluster.local:9092`), a
platform-wide constant, not per-row metadata (same category as Trino/
Postgres connection info never being per-feed metadata).

## 2. Get real messages flowing onto the topic **[Command]**

Schema discovery (next step) needs real messages already sitting in the
topic to sample from — there's no metadata that describes a stream's shape
ahead of time the way there is for `data_feed`. `inventory_events` has no
dedicated producer app (unlike `sales_events`, which gets a real
long-running `Deployment`, `streaming/producer/`) — for a genuinely new
source, you either write one (mirroring `streaming/producer/producer/main.py`)
or, for a one-off/manual source, produce a few real-shaped messages by
hand:

```bash
kubectl exec -n streaming deploy/kafka -- /opt/kafka/bin/kafka-topics.sh \
  --bootstrap-server localhost:9092 --create --if-not-exists \
  --topic inventory-events --partitions 1 --replication-factor 1

cat <<'EOF' > /tmp/inventory_events_sample.jsonl
{"event_id": "wt-0001", "sku": "SKU-2001", "branch": "A", "quantity_change": -4, "event_timestamp": "2026-07-19 09:00:00.000000"}
{"event_id": "wt-0002", "sku": "SKU-2002", "branch": "B", "quantity_change": 12, "event_timestamp": "2026-07-19 09:00:01.000000"}
{"event_id": "wt-0003", "sku": "SKU-2003", "branch": "C", "quantity_change": -2, "event_timestamp": "2026-07-19 09:00:02.000000"}
EOF
kubectl cp /tmp/inventory_events_sample.jsonl \
  streaming/"$(kubectl get pod -n streaming -l app=kafka -o jsonpath='{.items[0].metadata.name}')":/tmp/inventory_events_sample.jsonl
kubectl exec -n streaming deploy/kafka -- sh -c \
  '/opt/kafka/bin/kafka-console-producer.sh --bootstrap-server localhost:9092 --topic inventory-events < /tmp/inventory_events_sample.jsonl'
```

**`event_timestamp` must be the SQL-standard space-separated format**
(`'%Y-%m-%d %H:%M:%S.%f'`), not Python's default `.isoformat()`
(`'T'`-separated) — the Flink sink's `CAST(... AS TIMESTAMP(6))` fails
*silently* on the ISO form (a real gotcha, see `Learnings.md`, "Flink SQL's
`CAST(string AS TIMESTAMP(n))` fails silently on an ISO-8601
`'T'`-separated string"). `branch` values matching real `sales_dim_branch`
rows (`A`/`B`/`C`) are what make step 7's join resolve to real cities
instead of nulls.

A heredoc, not a shell loop with interpolated variables, for the same
reason `streaming/testing/run.py` produces messages this way — confirmed
live, a host shell expanding variables before they reach `kubectl exec`'s
nested `sh -c` can corrupt JSON into multi-line garbage (see `Learnings.md`).

## 3. Discover the schema **[UI]**

Streamlit → **Streaming Sources** → *Discover schema* → select
`inventory_events` → **Discover schema now**.

This connects to the topic, consumes up to the configured sample size of
real messages, infers column types (`connectors.inference.infer_column_definitions`,
the same schema-discovery code path a batch feed's extraction step uses),
and writes a new `schema_registry` version
(`controlling_object_type='streaming_source'`). You should see 5 columns
discovered: `event_id`, `sku`, `branch`, `quantity_change`,
`event_timestamp`.

## 4. Pick the event timestamp column **[UI]**

Streamlit → **Streaming Sources** → *Edit existing* → select
`inventory_events` → **Event timestamp column** → `event_timestamp` →
**Save changes**.

This is the one field schema discovery can't infer on its own — the Flink
sink's generated SQL needs to know which discovered column to
`CAST(... AS TIMESTAMP(6))`. `event_timestamp_column` is null until this
step runs; `scripts/generate_streaming_ingestion.py` (next step) skips a
row with a clear message, not an error, until both this and step 3 are
done.

## 5. Deploy the ingestion plumbing **[Command]**

```bash
just streaming::generate-streaming-ingestion
just kafka::start
just flink::start
```

Zero hand-written plumbing here — `generate-streaming-ingestion` reads
`inventory_events`'s `streaming_source` row + its current `schema_registry`
entry and renders the Flink SQL sink script
(`streaming/flink/sql-scripts/generated/inventory_events_sink.sql`), the
`FlinkDeployment` CR (`streaming/flink/generated/inventory-events-flinkdeployment.yaml`
— note the hyphen: Kubernetes resource names can't contain underscores,
`table_name` keeps them everywhere else), and the Kafka topic-creation Job.
`kafka::start`/`flink::start` apply them for real; `flink::start` polls
until `inventory-events-sink` reaches `RUNNING`.

## 6. Verify the raw data is flowing **[Command]**

```bash
kubectl exec -n query-engine deployment/trino-coordinator -- trino --execute \
  "SELECT count(*) FROM iceberg.streaming.inventory_events"
# produce a few more messages (step 2), wait ~15s for Flink's 10s checkpoint
# interval, re-run -- confirm the count increased
```

This confirms Kafka → Flink → Iceberg end to end, independent of any dbt
model — the Iceberg table exists because Flink's `CREATE TABLE IF NOT
EXISTS` created it, not dbt.

## 7. Write the business-logic serve view **[Code]**

`scripts/generate_streaming_serve_scaffolds.py` already wrote a starting
scaffold the first time `just orchestration::start` ran after step 1 —
`dbt/domains/sales/models/serve/streaming/inventory_events.sql`, a bare
`SELECT * FROM {{ source('streaming', 'inventory_events') }}` with a TODO
comment. Write-if-missing, never touched again once it exists — same
"boilerplate generated, business logic is yours" split as a batch fact's
join to its dimension (`generate_model_scaffolds.py`'s own reasoning: no
metadata describes which dimension a stream should join to or on what
key). Replace it with the real join:

```sql
{{ config(schema='serve', materialized='view', tags=['streaming', 'streaming_sales', 'streaming_inventory_events']) }}

select
    e.event_id,
    e.sku,
    e.branch,
    b.city,
    e.quantity_change,
    e.event_timestamp
from {{ source('streaming', 'inventory_events') }} e
left join {{ ref('sales_dim_branch') }} b
    on e.branch = b.branch
   and b.is_deleted = false
```

The three tags matter — `streaming` keeps this view out of the regular
batch `sales_modeling_job` build entirely (`dbt_assets.py`'s
`_STREAMING_TAG`; a streaming serve view racing Flink's first checkpoint
inside an ordinary batch run is a real bug this project hit once, see
`Learnings.md`), while `streaming_sales`/`streaming_inventory_events` give
per-domain/per-object selection for exactly this kind of standalone build.
Assumes `sales_dim_branch` already exists and is populated — streaming
serve views join *out* to already-persisted model-layer tables, they don't
populate them (batch-first design, confirmed directly with the user).

## 8. Build and verify the serve view **[Command]**

```bash
cd dbt/domains/sales
uv run dbt build --profiles-dir profiles --select tag:streaming_inventory_events
cd ../../..
```

```bash
kubectl exec -n query-engine deployment/trino-coordinator -- trino --execute \
  "SELECT event_id, sku, branch, city, quantity_change FROM iceberg.serve.inventory_events ORDER BY event_timestamp DESC LIMIT 5"
```

Every row should have a real `city` (`Yangon`/`Mandalay`/`Naypyitaw`), not
`NULL` — that's the join actually resolving against `sales_dim_branch`,
confirmed live, not just a view that compiles.

Streaming is this platform's smoke-test/dev-cycle module, not meant to be

```bash
just streaming::kill
```

## 10. (Alternative) Validate a streaming source without doing any of this by hand

`streaming/testing/` runs steps 2–8 above automatically, in isolation
(dummy model-layer fixtures instead of requiring a real batch run first) —
see `Progress.md`'s "Isolated streaming test suite" entry and
`streaming/DebugReference.md`. It's a hand-maintained test fixture
(`run.py`'s `_MESSAGE_FIXTURES`), not a substitute for this walkthrough —
it exists to prove the *platform* still works, not to onboard a *new*
source for a real user. A genuinely new streaming source always needs a
human to do steps 1–4 by hand, same as a genuinely new batch feed does.

```bash
just streaming-testing::test
```
