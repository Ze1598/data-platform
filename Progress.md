- [x] Single-node kind cluster config (`platform/kind/kind-cluster.yaml`) with `extraMounts` → `./data-lake`

Notes / deviations:
- `platform/kind/kind-cluster.yaml` also adds `extraPortMappings` (host `5432` → node `30432`) paired with a `NodePort` Postgres Service (not in the original Roadmap wording) — this keeps `localhost:5432` working unchanged for host-side tools (the Phase 1 Streamlit app, a local `psql` client) even though Postgres now runs in-cluster. Re-ran the full Phase 1 CRUD verification against the in-cluster Postgres with zero app changes required — confirms this decision worked.
- Roadmap's `platform/storage/` (PV/PVC + StorageClass) was **not** built as literal PV/PVC objects. A PersistentVolumeClaim binds 1:1 to one PV, but the data-lake needs to be readable/writable by many pods across many namespaces (Trino, Spark, Polaris, Dagster ops in later phases) — PVCs are also namespace-scoped, so they can't be shared across namespaces anyway. Used a direct `hostPath` volume (`/data-lake`, backed by the kind node's `extraMounts`) in each pod spec instead, which is the standard pattern for this exact scenario in local/kind clusters. Postgres's own data directory *is* a real single-consumer PVC, via `volumeClaimTemplates` on the StatefulSet (kind's built-in `standard` / local-path-provisioner StorageClass, no custom StorageClass needed).
- `scripts/bootstrap_kind.sh` generates the `postgres-init-scripts` ConfigMap from `metadata/db/init/` via `kubectl create configmap --from-file --dry-run=client -o yaml | kubectl apply -f -` rather than hand-duplicating the SQL into a static manifest, so `01_platform_metadata.sql` has one source of truth.
- Postgres credentials are a plaintext `Secret` (`metadata/k8s/secret.yaml`), matching `.env.example` — fine for a single-user local learning cluster, explicitly not how this would be done against a real cluster (noted in a comment in the file).
- The Phase 1 docker-compose Postgres was stopped (`docker compose down`) to free host port 5432 for the kind NodePort mapping. `docker-compose.yml` is left in place as a still-valid non-k8s quick-start path, but the in-cluster Postgres is now the source of truth going forward.

## Phase 3 — Apache Polaris + Trino, manual MERGE proof
- [x] `polaris_db` provisioned in the shared Postgres instance
- [x] Polaris deployed (Postgres-backed via `relational-jdbc`)
- [x] MinIO deployed (S3-compatible object storage, `lakehouse` bucket) — **not in the original plan**, added mid-phase after `FILE` storage type proved unworkable (see Notes)
- [x] Trino deployed with `iceberg.properties` (REST catalog config, S3/MinIO variant, legacy Hadoop-S3A filesystem)
- [x] `S3`-type Polaris catalog registered against MinIO (not the originally-planned `FILE` type)
- [x] Table created and seeded in `clean` via Trino (not hand-written Iceberg files — created through Trino SQL, which is what actually exercises the Trino+Polaris+MinIO write path)
- [x] **Verify**: manual `SELECT` and `MERGE INTO staging` succeed from Trino — confirmed correct upsert semantics (existing row updated, new row inserted, untouched rows left alone)

Notes / deviations:
- **`FILE` storage type abandoned.** Polaris 1.6.0's `FILE` storage type — the reason Polaris was originally chosen over Lakekeeper/Nessie/Unity Catalog OSS — turned out to be gated behind multiple layered "insecure, must not be used" checks: a hard startup-abort production-readiness check (`SUPPORTED_CATALOG_STORAGE_TYPES` containing `FILE`) and a separate request-time gate (`ALLOW_INSECURE_STORAGE_TYPES`) that together resisted every documented bypass tried (env vars, mounted `application.properties`, debug logging to trace the resolution). Pivoted to `S3` storage type backed by a self-hosted MinIO instance instead — fully local, no cloud account, and Polaris's well-supported, non-defense-gated path. `data-lake/clean`, `data-lake/staging`, `data-lake/model`, `data-lake/iceberg` (the local folders created for the `FILE`-storage plan) are now vestigial — actual Iceberg table data lives in MinIO's `lakehouse` bucket, not on the host filesystem.
- **New component**: `query-engine/minio/` (Secret, PVC, Deployment, Service, bucket-creation Job). MinIO is ClusterIP-only (no host port mapping needed) — only Polaris and Trino talk to it, and both are in-cluster.
- **Polaris + MinIO config gotchas** (see Roadmap.md "Iceberg-specific caveats" for the full list): `S3` storage type needs a dummy `roleArn` even for MinIO; `storageConfigInfo` fields (`pathStyleAccess`, `stsUnavailable`) only reliably apply at catalog **creation**, not via `PUT` update, and there's no CLI-supported update path either — confirmed still true in Phase 4 cleanup, delete+recreate remains correct for storage config changes. Polaris's credential-vending path for staged table creation (`CREATE_TABLE_STAGED_WITH_WRITE_DELEGATION`) and table purges (`DROP_TABLE_WITH_PURGE`) were rejected for every principal including root on this version — **resolved in Phase 4 cleanup**: both were a missing RBAC grant (`TABLE_WRITE_DATA`, only satisfied by `CATALOG_MANAGE_CONTENT`, which our `catalog_admin` role never had — traced via the exact `apache-polaris-1.6.0` authorization source, not assumption), not a deliberate CVE lockdown. Fixed via `polaris privileges catalog grant ... CATALOG_MANAGE_CONTENT`. Re-enabling `vended-credentials-enabled=true` on Trino still doesn't work after that fix, but for an unrelated reason — the catalog's `stsUnavailable: true` means Polaris has no real credentials to vend at all (`Credential vending was requested ..., but no credentials are available`) — so `vended-credentials-enabled=false` stays as permanent, correct config for a MinIO-backed catalog, not a pending-fix workaround. **The actual root cause of the core blocker**: Polaris's own server-side `S3FileIO` client (independent of anything Trino sends) wasn't picking up the catalog's `s3.endpoint` property and was defaulting to real AWS S3 — confirmed by reading Polaris's own logs and by capturing live MinIO traffic showing the failing request never arrived. Fixed by setting `AWS_ENDPOINT_URL_S3` (plus `AWS_REGION`/`AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`) directly as env vars on the **Polaris** deployment — a separate credential path from Trino's own catalog config.
- A suspected Trino bug (trinodb/trino#25187, path-style-access not honored with Iceberg REST + S3) turned out to be a red herring — the real fix was Polaris-side. Trino was left on the legacy Hadoop-S3A filesystem (`fs.hadoop.enabled` + `hive.s3.*`) rather than the modern `fs.native-s3` at the time, since that was the config already in place when the actual root cause was found. **Re-tested and reverted in Phase 4 cleanup**: switched back to `fs.native-s3` + `s3.*` properties, confirmed working (read, create, insert, select, full `dbt build` all green) — the legacy filesystem config is no longer used.
- One orphaned test table (`iceberg.clean.customers5`), previously undroppable due to the `DROP_TABLE_WITH_PURGE` permission gate above, was dropped in Phase 4 cleanup once that gate was resolved.

Notes / deviations:
- **Blocked, then resolved: Polaris `DROP_WITH_PURGE_ENABLED`.** dbt-trino's `merge` incremental strategy creates and drops a temp view (`customers__dbt_tmp`) every run; Polaris rejected the `DROP ... purge` with `Unable to purge entity ... set the Polaris configuration DROP_WITH_PURGE_ENABLED or the catalog configuration polaris.config.drop-with-purge.enabled`. Four env-var-based attempts at the realm-level `POLARIS_FEATURES_DROP_WITH_PURGE_ENABLED` flag all had zero effect — the error message names two *different* mechanisms, and only the second (`polaris.config.drop-with-purge.enabled`, a **per-catalog property** on the catalog entity itself, not a realm/Quarkus setting) actually applies once a catalog already exists. Setting it via a hand-rolled `curl PUT` to the Management API also silently failed to persist. **Fix**: set it via the official Polaris CLI (`apache-polaris` on PyPI, run via `uvx --from apache-polaris polaris ... catalogs update --set-property polaris.config.drop-with-purge.enabled=true data_platform`) — persistence confirmed via `catalogs get` (`entityVersion` incremented). This also revises the Phase 3 note that catalog `PUT` updates are fundamentally unreliable: in hindsight that was very likely hand-rolled `curl` requests not constructing the update request correctly (missing `currentEntityVersion`/full property merge), not a genuine Polaris server bug — the CLI updates a live catalog correctly. `register-catalog.sh` now includes `polaris.config.drop-with-purge.enabled` in the catalog's creation-time properties so a fresh bootstrap doesn't need the follow-up CLI step at all. Full writeup in [Learnings.md](Learnings.md).
- This resolves the `DROP_TABLE_WITH_PURGE` rough edge noted as "unresolved" in Phase 3 for dbt's use case (dropping the temp *view*, which only needs the catalog-level property fix above). It turned out there's a second, separate gate for purging a real *table* — a missing RBAC privilege grant — fixed separately during Phase 4 cleanup; see the Phase 3 notes above and Learnings.md.

### Mid-phase pivot: Polars-native refactor + schema-evolution system

While stress-testing `financial_transactions` at millions-of-rows scale, a real inconsistency got called out directly: a Polars-vectorized bulk CSV generator had been added *alongside* the original plain-`csv`-module small-batch generator, and `financial_assets.py`'s landing/raw/clean steps were still round-tripping through Python `list[dict]` instead of native Polars DataFrames — inconsistent, and specifically flagged as a decision that shouldn't have been made unilaterally. This became a full-scope correction: **all four feeds** (`customers`, `sales`, `financial_transactions`, `police_crimes`), not just the one under active stress-testing, moved to Polars-DataFrame-native asset I/O (`Output[pl.DataFrame]` landing→raw→clean throughout, no `list[dict]` anywhere in that path). `generate_financial_reports.py` collapsed to one Polars-vectorized generator parameterized by row count (25 for a normal drop, millions for a stress test), removing the plain-`csv`-module path entirely.

This pivot also produced a new standing rule (now a persistent memory): **always ask before making data-processing or metadata decisions**, rather than picking an approach and presenting the result — directly responsible for the multi-round design negotiation that produced the schema-evolution system below (eager vs. lazy DataFrames, empty-batch behavior, the generator-unification approach, the optional Spark-DDL-style schema override, and — critically — the exact three-way split on how a feed's schema is allowed to drift).

**New: reactive schema-registry reconciliation** (`processing/raw_to_clean/raw_to_clean/schema_evolution.py`, `reconcile_schema()`), replacing hardcoded per-feed `schema_overrides` dicts entirely:
- A registered column that's all-null this run gets cast to its already-known registered type (or `Utf8` if brand new) — not a schema change, no new information to act on.
- A genuinely new column (not yet in `schema_registry`) gets auto-added, nullable, at the next ordinal — schema evolves, run proceeds.
- A registered column whose *non-null* data now has a different concrete type than registered (e.g. an all-digit ID column inferring `Int64` where the registry says `string`) is treated as a legitimate upstream change — `PostgresMetadataResource.update_schema_registry()` (new: versioned insert, flips the old row's `is_current` before inserting the new one, respecting the `uq_schema_registry_current` partial unique index) writes a new version, and the run proceeds under it, it does not fail.
- A column present in the registry but **absent** from this run's data always fails the feed (`MissingColumnsError`) — the one case never auto-healed, since silently dropping a promised column would corrupt every downstream consumer. Scoped correctly to just that feed and its dependents by the existing per-feed pool/error-propagation machinery (`log_data_feed_stage` already logs-and-reraises) — no new isolation mechanism needed.
- An optional Spark-DDL-style schema override (`parse_ddl_schema()`, e.g. `"account_code string, amount double"`) is available for callers that want to force a type at construction time, using Polars' own type-name vocabulary directly (`getattr(pl, type_name)`) — no translation layer.

Two consequences this design surfaced, both handled:
- **Iceberg's schema evolution only allows narrow "safe" promotions in place** (`int`→`long`, `float`→`double`) — a `string`→`long` change (the real `account_code` case) isn't legal in-place. Since `clean` never retains cross-run history (a snapshot of just this run's batch, always overwritten), `write_clean_snapshot()` now takes a `schema_changed` flag and drops+recreates the table under the new schema when set, sidestepping Iceberg's promotion rules entirely rather than attempting in-place evolution with a fallback path.
- **A downstream cumulative dbt model still has the *old* type baked into its own materialized schema** even after `clean`'s schema correctly evolves — Trino's `MERGE` requires matching source/target types, so the very next incremental run fails (`MERGE table column types don't match`) even though the upstream change was itself handled correctly. Fixed at the dbt layer, not the ingestion layer: every `stg_<feed>.sql` model now explicitly casts each source column to a fixed, deliberately-chosen target type in its own `select`, decoupling staging's stable warehouse contract from whatever type `clean` happens to infer on any given run — standard dbt staging-layer practice, not something specific to this project's Polars/Iceberg stack.

All three schema-evolution scenarios verified for real against `financial_transactions`: a genuine `account_code` string→`long` evolution (plus, unplanned but instructive, a second real drift the same mechanism auto-healed — a stale `schema_registry` row from before an earlier uncommitted seed-script edit); a new `memo` column added mid-stream (auto-evolved, run succeeded, downstream `stg_financial_transactions.sql`'s explicit column list correctly ignored the new unreferenced column); and a deliberately-missing required column (`cost_center`/`memo` both absent from one test batch) correctly failing just `financial_transactions` while `customers`/`sales`/`police_crimes` all succeeded in the same `__ASSET_JOB` run.

### Post-completion refinement: SHA-512 hashes, no more MERGE, and a real OOM root-cause investigation

Prompted by a direct review of the Trino OOM incident above — three corrections, plus the investigation the user specifically asked for ("a better understanding of what exactly ran OOM").

- **`row_hash()` macro: MD5 → SHA-512.** One-line change (`macros/row_hashes.sql`), propagates to every `_key_hash`/`_attr_hash` across all 7 places the macro is used (4 staging models, `dim_branch`, `fct_sales`, `dim_customer_snapshot`). Confirmed post-change via a live query: 128-hex-char hashes (SHA-512), not 32 (MD5).
- **All 6 anti-join-gated models rewritten**: `stg_customers`, `stg_sales`, `stg_financial_transactions`, `stg_police_crimes`, `dim_branch`, `fct_sales` no longer use `incremental_strategy='merge'` at all. Each now computes a single join against its target (not a pre-filter join *plus* a second join inside a generated MERGE), classifying every surviving row as `'insert'` or `'update'` via an explicit `_change_type` column, applied through a project-level override of `trino__get_delete_insert_merge_sql` (`macros/insert_update_split.sql`) that does a targeted `DELETE` (for the update set) then `INSERT` (everything) — no `MERGE` statement anywhere in the project. `dim_customer_snapshot` (the Type 2 snapshot) needed no changes — dbt's native `check`-strategy snapshot machinery already does an equivalent two-statement UPDATE-then-INSERT internally, not a MERGE.
- **New `data_feed.updates_enabled` column** (mirrors `model_feed.updates_enabled`, which already existed but was never actually wired into any pipeline behavior before this). Threaded from Postgres into dbt via `--vars` (`PostgresMetadataResource.get_updates_enabled_map()`, called once per feed before `dbt build` in `dbt_assets.py`) — Trino has no catalog federating into `platform_metadata`, so this can't be a live lookup inside the SQL itself. When `false` for a given model, its attribute-hash comparison is omitted entirely at compile time and the feed is treated as insert-only.
- **The OOM root cause, investigated properly this time** (not just fixed by raising a limit): reverted Trino to its exact original undersized limits (2Gi container / 1500M heap / 700MB `maxMemoryPerNode`) and reran a comparable multi-million-row load under the new no-MERGE, single-join models. **It OOMKilled again** — this time on ~1.3M rows, fewer than the original 3M. This is decisive: the double-join was a real, worthwhile inefficiency to remove (and it's gone now), but it was not the dominant cause of the OOM. The dominant cause is structural: `server.workers: 0` (the Trino coordinator doing double duty as its only worker, deliberately, to fit a laptop-sized kind cluster) concentrates an entire multi-million-row query's memory footprint onto one node — no query-shape rewrite fixes that, only more memory (vertical scaling, the only option available here) or more nodes (horizontal, not available in this dev topology) does. Trino restored to 4Gi/3000M/1800MB; the same batch that OOMKilled at the old limit completed cleanly (confirmed twice: once mid-investigation, once as a final full-pipeline verification run with the restored sizing — `4/4` feeds, `4,847,916` rows in `staging.financial_transactions`, zero duplicates).
- Full technical detail, including the exact adapter-dispatch mechanism used to override MERGE without needing to patch dbt-trino, in Learnings.md.

### Second refinement pass: realizing updates_enabled's value, generalizing deletion synthesis, and deduplicating the 6 models

A follow-up tech-debt review surfaced three items worth acting on immediately rather than leaving as backlog.

- **`updates_enabled` values actually set, not just the mechanism built.** `financial_transactions` and `sales` are both real-world immutable once written (a posted GL entry gets a reversing entry, not an edit; a completed invoice line gets refunded/voided, not rewritten) — flipped both to `updates_enabled=false` in `seed_metadata_db.py`. `customers` and `police_crimes` stay `true` (customers genuinely gets updated; a crime's `outcome_category`/`outcome_date` genuinely change after the fact). Verified post-flip: `staging.sales` at 20/20 distinct keys, no duplicates.
- **Deletion synthesis generalized from one hand-written model to codegen.** `int_customers_with_deletes.sql` was entirely hand-written and column-specific (hardcoded `customer_id, name, email, updated_at`). New `scripts/generate_deletion_synthesis_views.py` (same shape as Phase 8's `generate_serve_views.py`) reads every `deletions_enabled=true` `model_feed` row plus its `schema_registry`-current column list, and renders `models/model/dimensions/generated/int_<feed>_with_deletes.sql` per feed — column lists come from `schema_registry`, not live Trino introspection, keeping this resolvable from Postgres alone at build time like every other codegen step in this project. Confirmed the generated output for `customers` is logically byte-identical to the original hand-written model. Wired into `orchestration::start` alongside `generate-serve-views`.
- **The 6 rewritten models deduplicated.** Each had hand-written nearly-identical join + `_change_type` classification logic. Extracted into a new `classify_changes(source_relation, updates_enabled)` macro (`macros/insert_update_split.sql`, alongside the `trino__get_delete_insert_merge_sql` override it pairs with) — every model's incremental branch is now a one-line macro call instead of a ~12-line hand-rolled CTE.
- **Verified together in one real run**: full `__ASSET_JOB` (all 4 feeds) succeeded end-to-end (`4/4` + `4/4`) in 1m23s, `dim_customer_snapshot` deletion synthesis confirmed unchanged (5 active customers, 0 deleted, matching baseline), `sales` confirmed insert-only with zero duplicates, and the full test suite (44 dbt tests + every pytest suite) green afterward.

### Third refinement pass: real `landing`/`raw`/`archive` semantics for `financial_transactions`

A design correction, not an incremental patch: `raw` was never actually implemented on any feed (every `raw_<feed>` asset was a pure in-memory passthrough since Phase 5/6), which is exactly why `landing` had been silently accumulating every file ever dropped, forever — there was no durable copy anywhere else to justify clearing it. Corrected the model for `financial_transactions` (the only feed with real files sitting in `landing`):

- **`landing`** is the drop zone, conceptually external to the platform's own storage (matches the original Roadmap wording). Read-only from the pipeline's perspective except for the final wipe step.
- **`raw`** (`raw_financial_transactions`, now a real file write, not a passthrough) makes a verbatim, byte-for-byte copy of whatever's currently in `landing` into `data-lake/raw/financial_transactions/run_id=<dagster_run_id>/` — no transformation, `clean` is still where parsing/typing happens. This run-scoped folder becomes the single source of truth for what the run consumed.
- **`archive`** (new `archive_financial_transactions` asset) copies that same run-scoped `raw` folder into a watermark-style timestamped location (`data-lake/archive/financial_transactions/YYYY/MM/DD/HH/MM/SS/`, one folder per run) — the long-term disaster-recovery copy; reload archived folders in run order to rebuild raw/clean/staging from scratch if ever needed.
- **`landing` gets wiped**, file by file, only as each one is successfully archived — and `archive_financial_transactions` only runs at all if `dbt_financial_transactions_assets` (the whole feed's dbt build, through the model layer) fully succeeded, via a plain `deps=[...]` dependency. A failure anywhere upstream (raw copy, clean, dbt build) means this asset never executes, so `landing` is left completely untouched for a retry — no separate error handling needed to guarantee that, Dagster's own skip-on-upstream-failure behavior provides it for free.
- **Verified end-to-end**: a real run correctly copied the landing file to `raw/.../run_id=<id>/`, then to `archive/2026/07/11/21/07/23/`, then deleted it from `landing` — confirmed via direct filesystem inspection at every step, not just asset success. A second full-pipeline run on now-empty `landing` correctly no-op'd (`archive_financial_transactions` recognizes an absent/empty raw run-folder and skips cleanly). Full `just test` green afterward (44/44 + every pytest suite).
- **Deliberately scoped to `financial_transactions` only** — the sole feed with real files sitting in `landing` today; `customers`/`sales` are in-memory generated stubs and `police_crimes` is API-sourced, neither has anything in `landing` to copy/archive/wipe.

### Dynamic pipeline foundation, real schedules, multi-feed model

Asking to test `schedule` functionality surfaced that there was no dynamic, metadata-driven pipeline at all — every feed's jobs were hand-written Python, only 2 of 4 feeds even had a dedicated triggerable job, and `police_crimes_schedule`'s cron was a hardcoded constant. Full addendum plan in `.claude/plans/i-want-to-create-calm-manatee.md`; approved and implemented in full.

- **Every feed now gets a real, generated job** (`customers_job`/`sales_job`/`financial_transactions_job`/`police_crimes_job`), replacing the two hand-written ones and the `__ASSET_JOB`-only reliance for `customers`/`sales`. Driven by `group_name=<feed friendly_name>` on every asset (hand-written + dbt, via a new `DataPlatformDbtTranslator.get_group_name()` override) and a new build-time codegen script, `scripts/generate_dagster_pipeline.py` → `pipeline_generated.py` (same "structural decisions are codegen, never a live Postgres read at import time" rule as `generate_serve_views.py`).
- **`schedule` rows become real Dagster schedules**: a feed-type row → one generated `@schedule`; a model-type row → one generated schedule *per feed in that model's `depends_on_feeds`* (a schedule binds to exactly one job, and models have no standalone job). Each schedule's execution function re-checks `is_active` and looks up the target's current `batch_group_friendly_name` live at every tick (`PostgresMetadataResource.is_schedule_active()`) — structure is build-time, values are runtime, matching the user's stated preference.
- **New multi-feed test model**: `fct_daily_financial_activity` (row-grain `UNION ALL` of `stg_sales` + `stg_financial_transactions`, keyed by `(source_feed, source_id)`), `depends_on_feeds=[sales, financial_transactions]`. First lakehouse_models row depending on `financial_transactions` — flips `stg_financial_transactions` from the zero-dependents default (`updates_enabled=true`) to `false`, matching that staging model's own already-stated insert-only assumption (correction, not regression).
- **`customers`/`sales` raw semantics brought in line** with `financial_transactions`/`police_crimes` — `raw_customers`/`raw_sales` now write a durable parquet file to `raw/<feed>/run_id=<id>/`, no archive step (synthetic smoketest data, no retention need).
- **Real bugs found and fixed during implementation, not just design-time**:
  1. A dbt model tagged with two feed tags gets claimed by *two* separate `@dbt_assets` Python defs, which Dagster rejects as a duplicate AssetKey — hit for real on both the base model and the derived serve views (`generate_serve_views.py` was independently tagging with every `depends_on_feeds` member). Fixed by giving every generated per-feed job's `AssetSelection` `.upstream()` (not a bare `.groups(feed)`) so a feed's job still pulls in genuine cross-feed Dagster dependencies, and by making both the base model's tag and `generate_serve_views.py`'s derived tag agree on a single deterministic choice (alphabetically-first `depends_on_feeds` member).
  2. A `@schedule` function requesting a resource must declare it as a named parameter (`def _fn(context, postgres_metadata: ...)`), not pull it from `context.resources` — `ScheduleDefinition.__call__`'s resource-injection matches by parameter name, and passing `required_resource_keys=` as well is rejected outright ("cannot specify resource requirements in both").
  3. A GraphQL-submitted run only queues — the daemon creates the actual k8s Job a moment later, same asynchronous gap `verify-pipeline`'s job-name-diffing loop already existed for. `verify-schedule` needed the same appear-then-wait pattern, not a direct `kubectl wait`.
- **Verified live**: full `just smoketest` green (`verify-pipeline` 4/4+4/4 plus new durable-raw-file assertions, new `verify-schedule` recipe launching a real schedule-evaluated run end-to-end via `dagster_graphql.DagsterGraphQLClient`, 54/54 dbt tests including the new model, 5/5 orchestration tests including 4 new schedule tests, full pytest suite).
- **Explicit backlog note, not implemented**: CRUD for `schedule` rows in the frontend, per the existing `2_Data_Feeds.py`/`3_Lakehouse_Models.py` pattern.
