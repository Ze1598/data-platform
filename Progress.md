- [x] Single-node kind cluster config (`platform/kind/kind-cluster.yaml`) with `extraMounts` → `./data-lake`
- [x] Namespaces created: `metadata`, `orchestration`, `processing`, `query-engine`, `frontend`

Notes / deviations:
- `platform/kind/kind-cluster.yaml` also adds `extraPortMappings` (host `5432` → node `30432`) paired with a `NodePort` Postgres Service (not in the original Roadmap wording) — this keeps `localhost:5432` working unchanged for host-side tools (the Phase 1 Streamlit app, a local `psql` client) even though Postgres now runs in-cluster. Re-ran the full Phase 1 CRUD verification against the in-cluster Postgres with zero app changes required — confirms this decision worked.
- Roadmap's `platform/storage/` (PV/PVC + StorageClass) was **not** built as literal PV/PVC objects. A PersistentVolumeClaim binds 1:1 to one PV, but the data-lake needs to be readable/writable by many pods across many namespaces (Trino, Spark, Polaris, Dagster ops in later phases) — PVCs are also namespace-scoped, so they can't be shared across namespaces anyway. Used a direct `hostPath` volume (`/data-lake`, backed by the kind node's `extraMounts`) in each pod spec instead, which is the standard pattern for this exact scenario in local/kind clusters. Postgres's own data directory *is* a real single-consumer PVC, via `volumeClaimTemplates` on the StatefulSet (kind's built-in `standard` / local-path-provisioner StorageClass, no custom StorageClass needed).
- `scripts/bootstrap_kind.sh` generates the `postgres-init-scripts` ConfigMap from `metadata/db/init/` via `kubectl create configmap --from-file --dry-run=client -o yaml | kubectl apply -f -` rather than hand-duplicating the SQL into a static manifest, so `01_platform_metadata.sql` has one source of truth.
- Postgres credentials are a plaintext `Secret` (`metadata/k8s/secret.yaml`), matching `.env.example` — fine for a single-user local learning cluster, explicitly not how this would be done against a real cluster (noted in a comment in the file).
- The Phase 1 docker-compose Postgres was stopped (`docker compose down`) to free host port 5432 for the kind NodePort mapping. `docker-compose.yml` is left in place as a still-valid non-k8s quick-start path, but the in-cluster Postgres is now the source of truth going forward.

## Phase 3 — Apache Polaris + Trino, manual MERGE proof
- [x] `polaris_db` provisioned in the shared Postgres instance
- [x] Polaris deployed (Postgres-backed via `relational-jdbc`)
- [x] MinIO deployed (S3-compatible object storage, `lakehouse` bucket) — **not in the original plan**, added mid-phase after `FILE` storage type proved unworkable (see Notes)
- [x] Trino deployed with `iceberg.properties` (REST catalog config, S3/MinIO variant, legacy Hadoop-S3A filesystem)
- [x] `S3`-type Polaris catalog registered against MinIO (not the originally-planned `FILE` type)
- [x] Table created and seeded in `clean` via Trino (not hand-written Iceberg files — created through Trino SQL, which is what actually exercises the Trino+Polaris+MinIO write path)
- [x] **Verify**: manual `SELECT` and `MERGE INTO staging` succeed from Trino — confirmed correct upsert semantics (existing row updated, new row inserted, untouched rows left alone)

Notes / deviations:
- **`FILE` storage type abandoned.** Polaris 1.6.0's `FILE` storage type — the reason Polaris was originally chosen over Lakekeeper/Nessie/Unity Catalog OSS — turned out to be gated behind multiple layered "insecure, must not be used" checks: a hard startup-abort production-readiness check (`SUPPORTED_CATALOG_STORAGE_TYPES` containing `FILE`) and a separate request-time gate (`ALLOW_INSECURE_STORAGE_TYPES`) that together resisted every documented bypass tried (env vars, mounted `application.properties`, debug logging to trace the resolution). Pivoted to `S3` storage type backed by a self-hosted MinIO instance instead — fully local, no cloud account, and Polaris's well-supported, non-defense-gated path. `data-lake/clean`, `data-lake/staging`, `data-lake/model`, `data-lake/iceberg` (the local folders created for the `FILE`-storage plan) are now vestigial — actual Iceberg table data lives in MinIO's `lakehouse` bucket, not on the host filesystem.
- **New component**: `query-engine/minio/` (Secret, PVC, Deployment, Service, bucket-creation Job). MinIO is ClusterIP-only (no host port mapping needed) — only Polaris and Trino talk to it, and both are in-cluster.
- **Polaris + MinIO config gotchas** (see README.md "Iceberg-specific caveats" for the full list): `S3` storage type needs a dummy `roleArn` even for MinIO; `storageConfigInfo` fields (`pathStyleAccess`, `stsUnavailable`) only reliably apply at catalog **creation**, not via `PUT` update, and there's no CLI-supported update path either — confirmed still true in Phase 4 cleanup, delete+recreate remains correct for storage config changes. Polaris's credential-vending path for staged table creation (`CREATE_TABLE_STAGED_WITH_WRITE_DELEGATION`) and table purges (`DROP_TABLE_WITH_PURGE`) were rejected for every principal including root on this version — **resolved in Phase 4 cleanup**: both were a missing RBAC grant (`TABLE_WRITE_DATA`, only satisfied by `CATALOG_MANAGE_CONTENT`, which our `catalog_admin` role never had — traced via the exact `apache-polaris-1.6.0` authorization source, not assumption), not a deliberate CVE lockdown. Fixed via `polaris privileges catalog grant ... CATALOG_MANAGE_CONTENT`. Re-enabling `vended-credentials-enabled=true` on Trino still doesn't work after that fix, but for an unrelated reason — the catalog's `stsUnavailable: true` means Polaris has no real credentials to vend at all (`Credential vending was requested ..., but no credentials are available`) — so `vended-credentials-enabled=false` stays as permanent, correct config for a MinIO-backed catalog, not a pending-fix workaround. **The actual root cause of the core blocker**: Polaris's own server-side `S3FileIO` client (independent of anything Trino sends) wasn't picking up the catalog's `s3.endpoint` property and was defaulting to real AWS S3 — confirmed by reading Polaris's own logs and by capturing live MinIO traffic showing the failing request never arrived. Fixed by setting `AWS_ENDPOINT_URL_S3` (plus `AWS_REGION`/`AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`) directly as env vars on the **Polaris** deployment — a separate credential path from Trino's own catalog config.
- A suspected Trino bug (trinodb/trino#25187, path-style-access not honored with Iceberg REST + S3) turned out to be a red herring — the real fix was Polaris-side. Trino was left on the legacy Hadoop-S3A filesystem (`fs.hadoop.enabled` + `hive.s3.*`) rather than the modern `fs.native-s3` at the time, since that was the config already in place when the actual root cause was found. **Re-tested and reverted in Phase 4 cleanup**: switched back to `fs.native-s3` + `s3.*` properties, confirmed working (read, create, insert, select, full `dbt build` all green) — the legacy filesystem config is no longer used.
- One orphaned test table (`iceberg.clean.customers5`), previously undroppable due to the `DROP_TABLE_WITH_PURGE` permission gate above, was dropped in Phase 4 cleanup once that gate was resolved.

Notes / deviations:
- **Blocked, then resolved: Polaris `DROP_WITH_PURGE_ENABLED`.** dbt-trino's `merge` incremental strategy creates and drops a temp view (`customers__dbt_tmp`) every run; Polaris rejected the `DROP ... purge` with `Unable to purge entity ... set the Polaris configuration DROP_WITH_PURGE_ENABLED or the catalog configuration polaris.config.drop-with-purge.enabled`. Four env-var-based attempts at the realm-level `POLARIS_FEATURES_DROP_WITH_PURGE_ENABLED` flag all had zero effect — the error message names two *different* mechanisms, and only the second (`polaris.config.drop-with-purge.enabled`, a **per-catalog property** on the catalog entity itself, not a realm/Quarkus setting) actually applies once a catalog already exists. Setting it via a hand-rolled `curl PUT` to the Management API also silently failed to persist. **Fix**: set it via the official Polaris CLI (`apache-polaris` on PyPI, run via `uvx --from apache-polaris polaris ... catalogs update --set-property polaris.config.drop-with-purge.enabled=true data_platform`) — persistence confirmed via `catalogs get` (`entityVersion` incremented). This also revises the Phase 3 note that catalog `PUT` updates are fundamentally unreliable: in hindsight that was very likely hand-rolled `curl` requests not constructing the update request correctly (missing `currentEntityVersion`/full property merge), not a genuine Polaris server bug — the CLI updates a live catalog correctly. `register-catalog.sh` now includes `polaris.config.drop-with-purge.enabled` in the catalog's creation-time properties so a fresh bootstrap doesn't need the follow-up CLI step at all. Full writeup in [Learnings.md](Learnings.md).
- This resolves the `DROP_TABLE_WITH_PURGE` rough edge noted as "unresolved" in Phase 3 for dbt's use case (dropping the temp *view*, which only needs the catalog-level property fix above). It turned out there's a second, separate gate for purging a real *table* — a missing RBAC privilege grant — fixed separately during Phase 4 cleanup; see the Phase 3 notes above and Learnings.md.

### Mid-phase pivot: Polars-native refactor + schema-evolution system

While stress-testing `financial_transactions` at millions-of-rows scale, a real inconsistency got called out directly: a Polars-vectorized bulk CSV generator had been added *alongside* the original plain-`csv`-module small-batch generator, and `financial_assets.py`'s landing/raw/clean steps were still round-tripping through Python `list[dict]` instead of native Polars DataFrames — inconsistent, and specifically flagged as a decision that shouldn't have been made unilaterally. This became a full-scope correction: **all four feeds** (`customers`, `sales`, `financial_transactions`, `police_crimes`), not just the one under active stress-testing, moved to Polars-DataFrame-native asset I/O (`Output[pl.DataFrame]` landing→raw→clean throughout, no `list[dict]` anywhere in that path). `generate_financial_reports.py` collapsed to one Polars-vectorized generator parameterized by row count (25 for a normal drop, millions for a stress test), removing the plain-`csv`-module path entirely.

This pivot also produced a new standing rule (now a persistent memory): **always ask before making data-processing or metadata decisions**, rather than picking an approach and presenting the result — directly responsible for the multi-round design negotiation that produced the schema-evolution system below (eager vs. lazy DataFrames, empty-batch behavior, the generator-unification approach, the optional Spark-DDL-style schema override, and — critically — the exact three-way split on how a feed's schema is allowed to drift).

**New: reactive schema-registry reconciliation** (`processing/raw_to_clean/raw_to_clean/schema_evolution.py`, `reconcile_schema()`), replacing hardcoded per-feed `schema_overrides` dicts entirely:
- A registered column that's all-null this run gets cast to its already-known registered type (or `Utf8` if brand new) — not a schema change, no new information to act on.
- A genuinely new column (not yet in `schema_registry`) gets auto-added, nullable, at the next ordinal — schema evolves, run proceeds.
- A registered column whose *non-null* data now has a different concrete type than registered (e.g. an all-digit ID column inferring `Int64` where the registry says `string`) is treated as a legitimate upstream change — `PostgresMetadataResource.update_schema_registry()` (new: versioned insert, flips the old row's `is_current` before inserting the new one, respecting the `uq_schema_registry_current` partial unique index) writes a new version, and the run proceeds under it, it does not fail.
- A column present in the registry but **absent** from this run's data always fails the feed (`MissingColumnsError`) — the one case never auto-healed, since silently dropping a promised column would corrupt every downstream consumer. Scoped correctly to just that feed and its dependents by the existing per-feed pool/error-propagation machinery (`log_data_feed_stage` already logs-and-reraises) — no new isolation mechanism needed.
- An optional Spark-DDL-style schema override (`parse_ddl_schema()`, e.g. `"account_code string, amount double"`) is available for callers that want to force a type at construction time, using Polars' own type-name vocabulary directly (`getattr(pl, type_name)`) — no translation layer.

Two consequences this design surfaced, both handled:
- **Iceberg's schema evolution only allows narrow "safe" promotions in place** (`int`→`long`, `float`→`double`) — a `string`→`long` change (the real `account_code` case) isn't legal in-place. Since `clean` never retains cross-run history (a snapshot of just this run's batch, always overwritten), `write_clean_snapshot()` now takes a `schema_changed` flag and drops+recreates the table under the new schema when set, sidestepping Iceberg's promotion rules entirely rather than attempting in-place evolution with a fallback path.
- **A downstream cumulative dbt model still has the *old* type baked into its own materialized schema** even after `clean`'s schema correctly evolves — Trino's `MERGE` requires matching source/target types, so the very next incremental run fails (`MERGE table column types don't match`) even though the upstream change was itself handled correctly. Fixed at the dbt layer, not the ingestion layer: every `stg_<feed>.sql` model now explicitly casts each source column to a fixed, deliberately-chosen target type in its own `select`, decoupling staging's stable warehouse contract from whatever type `clean` happens to infer on any given run — standard dbt staging-layer practice, not something specific to this project's Polars/Iceberg stack.

All three schema-evolution scenarios verified for real against `financial_transactions`: a genuine `account_code` string→`long` evolution (plus, unplanned but instructive, a second real drift the same mechanism auto-healed — a stale `schema_registry` row from before an earlier uncommitted seed-script edit); a new `memo` column added mid-stream (auto-evolved, run succeeded, downstream `stg_financial_transactions.sql`'s explicit column list correctly ignored the new unreferenced column); and a deliberately-missing required column (`cost_center`/`memo` both absent from one test batch) correctly failing just `financial_transactions` while `customers`/`sales`/`police_crimes` all succeeded in the same `__ASSET_JOB` run.

### Post-completion refinement: SHA-512 hashes, no more MERGE, and a real OOM root-cause investigation

Prompted by a direct review of the Trino OOM incident above — three corrections, plus the investigation the user specifically asked for ("a better understanding of what exactly ran OOM").

- **`row_hash()` macro: MD5 → SHA-512.** One-line change (`macros/row_hashes.sql`), propagates to every `_key_hash`/`_attr_hash` across all 7 places the macro is used (4 staging models, `dim_branch`, `fct_sales`, `dim_customer_snapshot`). Confirmed post-change via a live query: 128-hex-char hashes (SHA-512), not 32 (MD5).
- **All 6 anti-join-gated models rewritten**: `stg_customers`, `stg_sales`, `stg_financial_transactions`, `stg_police_crimes`, `dim_branch`, `fct_sales` no longer use `incremental_strategy='merge'` at all. Each now computes a single join against its target (not a pre-filter join *plus* a second join inside a generated MERGE), classifying every surviving row as `'insert'` or `'update'` via an explicit `_change_type` column, applied through a project-level override of `trino__get_delete_insert_merge_sql` (`macros/insert_update_split.sql`) that does a targeted `DELETE` (for the update set) then `INSERT` (everything) — no `MERGE` statement anywhere in the project. `dim_customer_snapshot` (the Type 2 snapshot) needed no changes — dbt's native `check`-strategy snapshot machinery already does an equivalent two-statement UPDATE-then-INSERT internally, not a MERGE.
- **New `data_feed.updates_enabled` column** (mirrors `model_feed.updates_enabled`, which already existed but was never actually wired into any pipeline behavior before this). Threaded from Postgres into dbt via `--vars` (`PostgresMetadataResource.get_updates_enabled_map()`, called once per feed before `dbt build` in `dbt_assets.py`) — Trino has no catalog federating into `platform_metadata`, so this can't be a live lookup inside the SQL itself. When `false` for a given model, its attribute-hash comparison is omitted entirely at compile time and the feed is treated as insert-only.
- **The OOM root cause, investigated properly this time** (not just fixed by raising a limit): reverted Trino to its exact original undersized limits (2Gi container / 1500M heap / 700MB `maxMemoryPerNode`) and reran a comparable multi-million-row load under the new no-MERGE, single-join models. **It OOMKilled again** — this time on ~1.3M rows, fewer than the original 3M. This is decisive: the double-join was a real, worthwhile inefficiency to remove (and it's gone now), but it was not the dominant cause of the OOM. The dominant cause is structural: `server.workers: 0` (the Trino coordinator doing double duty as its only worker, deliberately, to fit a laptop-sized kind cluster) concentrates an entire multi-million-row query's memory footprint onto one node — no query-shape rewrite fixes that, only more memory (vertical scaling, the only option available here) or more nodes (horizontal, not available in this dev topology) does. Trino restored to 4Gi/3000M/1800MB; the same batch that OOMKilled at the old limit completed cleanly (confirmed twice: once mid-investigation, once as a final full-pipeline verification run with the restored sizing — `4/4` feeds, `4,847,916` rows in `staging.financial_transactions`, zero duplicates).
- Full technical detail, including the exact adapter-dispatch mechanism used to override MERGE without needing to patch dbt-trino, in Learnings.md.

### Third refinement pass: real `landing`/`raw`/`archive` semantics for `financial_transactions`

A design correction, not an incremental patch: `raw` was never actually implemented on any feed (every `raw_<feed>` asset was a pure in-memory passthrough since Phase 5/6), which is exactly why `landing` had been silently accumulating every file ever dropped, forever — there was no durable copy anywhere else to justify clearing it. Corrected the model for `financial_transactions` (the only feed with real files sitting in `landing`):

- **`landing`** is the drop zone, conceptually external to the platform's own storage (matches the original Roadmap wording). Read-only from the pipeline's perspective except for the final wipe step.
- **`raw`** (`raw_financial_transactions`, now a real file write, not a passthrough) makes a verbatim, byte-for-byte copy of whatever's currently in `landing` into `data-lake/raw/financial_transactions/run_id=<dagster_run_id>/` — no transformation, `clean` is still where parsing/typing happens. This run-scoped folder becomes the single source of truth for what the run consumed.
- **`archive`** (new `archive_financial_transactions` asset) copies that same run-scoped `raw` folder into a watermark-style timestamped location (`data-lake/archive/financial_transactions/YYYY/MM/DD/HH/MM/SS/`, one folder per run) — the long-term disaster-recovery copy; reload archived folders in run order to rebuild raw/clean/staging from scratch if ever needed.
- **`landing` gets wiped**, file by file, only as each one is successfully archived — and `archive_financial_transactions` only runs at all if `dbt_financial_transactions_assets` (the whole feed's dbt build, through the model layer) fully succeeded, via a plain `deps=[...]` dependency. A failure anywhere upstream (raw copy, clean, dbt build) means this asset never executes, so `landing` is left completely untouched for a retry — no separate error handling needed to guarantee that, Dagster's own skip-on-upstream-failure behavior provides it for free.
- **Verified end-to-end**: a real run correctly copied the landing file to `raw/.../run_id=<id>/`, then to `archive/2026/07/11/21/07/23/`, then deleted it from `landing` — confirmed via direct filesystem inspection at every step, not just asset success. A second full-pipeline run on now-empty `landing` correctly no-op'd (`archive_financial_transactions` recognizes an absent/empty raw run-folder and skips cleanly). Full `just test` green afterward (44/44 + every pytest suite).
- **Deliberately scoped to `financial_transactions` only** — the sole feed with real files sitting in `landing` today; `customers`/`sales` are in-memory generated stubs and `police_crimes` is API-sourced, neither has anything in `landing` to copy/archive/wipe.

**Part 3 — dbt-side boilerplate reduction (`scripts/generate_model_scaffolds.py`).** Closed the last open Phase 13 item. ### `schema_registry` ownership fix (2026-07-16)

Design settled in a dedicated implementation plan written in a prior session specifically so the fix wasn't designed in the same session that misdiagnosed it three times in a row (that plan file has been removed now that this work is done and fully captured here); implemented and verified in this one, with two real corrections made directly by the user during implementation (not guessed) — see `Learnings.md` for the durable lessons.

- **The rule, stated plainly**: schema discovery (and primary-key resolution) is exclusively the extraction step's job, complete before `clean` ever runs; `clean` only ever reads `schema_registry`, never writes it; `schema_registry` is never hand-seeded — a from-scratch feed or platform is expected to have zero rows there until that feed's first real run.
- **Every connector-driven feed's extraction step now owns discovery + `sync_schema_registry()`** (`connectors/schema_registry_sync.py`'s I/O side, `PostgresMetadataResource.sync_schema_registry()`), uniformly — this was already correct for tabular kinds (`postgres`/`csv`) but not for REST/JSON kinds, whose generated `clean_<feed>` bundled flatten+discover+sync together with validation, gated only on the `"validation"` pipeline step (meaning discovery silently never ran for a feed cherry-picked to extraction-only, and `clean` wasn't read-only the way the rule requires).
- **REST/JSON kinds' fix, not a straight port of the tabular shape**: flattening is what makes schema discovery possible for nested-JSON sources at all (`connectors/base.py`'s `JsonConnector`) — rather than flattening twice (once to discover, once to write), the extraction step now also performs the `clean`-layer write itself, reusing the one flattened DataFrame. `clean_<feed>` becomes a pure pass-through for these kinds, kept only so `clean.<feed>` has a stable `AssetKey` for dbt source lineage — there is no remaining validation-stage work for this connector shape. `raw_<feed>`'s own contract (verbatim, zero transformation) is completely unaffected — it still persists exactly the untouched nested fetch result it always did.
- **`customers`/`sales` (hand-written, predate the connector library) had the identical violation**, not just the codegen'd feeds — `clean_customers`/`clean_sales` also called `sync_schema_registry()` inline. Same fix applied (tabular shape, since these are flat synthetic generators, not nested).
- **`seed_schema_registry()` deleted outright**, along with its 4 call sites and the 4 dead hand-typed schema constants (`CUSTOMERS_SCHEMA`/`SALES_SCHEMA`/`FINANCIAL_TRANSACTIONS_SCHEMA`/`POLICE_CRIMES_SCHEMA`) — `metadata_runs` already had no seed row and bootstrapped correctly from a real run; every feed now follows that same pattern.
- **Terminology cleanup, by explicit direction**: the Dagster asset historically named `landing_<feed>` was renamed to `extraction_<feed>` everywhere (codegen templates + `extraction_assets.py`/`sales_assets.py` + `definitions.py`) — it never actually wrote to the `landing` storage layer, it's the extraction step's implementation (fetch from whatever the real source is); "landing" is only a meaningful concept when a feed's actual source *is* a landing directory (CSV/json_file kinds), not a generic pipeline-stage name. `data_processing_runs`' `landing_*` stage-tracking columns and the literal `data-lake/landing/` directory paths were deliberately left unrenamed — both are correctly named already, out of scope for this cleanup.
- **Second real bug, found during this fix's own live verification, not a regression from it and not in the original plan's scope**: `scripts/generate_deletion_synthesis_views.py` (generates `int_<feed>_with_deletes.sql`, the deletion-synthesis intermediate model) had its own unwarranted `schema_registry` dependency — it hard-crashed `just start` on a genuinely fresh platform, since `schema_registry` is correctly empty pre-first-run. Root-caused by reading the actual downstream consumer (`dbt/domains/sales/snapshots/sales_dim_customer.sql`, hand-written): it already names its own exact columns explicitly when selecting from this generated model, exactly the same way it already does selecting straight from staging — so the generated model never needed to enumerate columns from `schema_registry` at all. Fixed by switching its three `select <enumerated columns>` projections to `select *`, and deriving the join/anti-join predicate from `lakehouse_models.business_key_columns` (already-available metadata, no `schema_registry` involved) — removing the dependency entirely, not just tolerating a missing row. This also makes the "auto-generate all boilerplate the instant a user enters metadata" property strictly true for this model, where before it silently depended on a live run having already happened.
- **A doc/code drift found and fixed in passing**: this same script's output directory had drifted to `models/model/dimensions/generated/`; corrected to `models/model/intermediate/generated/`, matching README.md's own documented repo structure (`int_` is dbt's standard "intermediate model" naming convention — a reusable transformation step between staging and the final mart, neither raw staging nor itself the end-product dimension/fact). `.gitignore` updated to match; the stale pre-fix generated file removed.
- **Verified live**: full `just smoketest` (genuine nuke, rebuild) green — `verify-pipeline` 5/5 feed-run + 3/3 domain model-run, `verify-schedule` (real schedule-triggered run through the full webserver → daemon → `K8sRunLauncher` path) succeeded, 49/49 dbt tests, all pytest suites (`orchestration`, `frontend`, `tests/integration`). Directly confirmed via Postgres: every feed's current `schema_registry` row now has `created_by='extraction_<feed>'` (not a seed script), and `police_crimes` correctly resolved `primary_key_columns=["id"]` (previously silently empty due to the seed script never setting it) — the original ODS `COLUMN_NOT_FOUND` failure mode is gone, confirmed by regenerating ODS models twice in a row and getting byte-identical, keyed output both times.
- Hit the project's known `.pth`/`UF_HIDDEN` race once mid-verification (`dagster dev`'s long-running code server loaded `definitions.py` at exactly the wrong moment; short-lived `dagster job launch` subprocesses right next to it succeeded, since they got lucky on timing) — fixed per the standing rule (`rm -rf .venv && uv cache clean && uv sync --all-packages`, no diagnosis needed), then reran the full `smoketest` cycle clean.

## Cooperative wake-up mechanism for Dagster from Streamlit (2026-07-20) — built and live-verified

Closes the Backlog.md item of the same name: `frontend/pages/5_Trigger_Pipeline.py` now wakes `orchestration` itself when KEDA has scaled `dagster-webserver`/`dagster-code-server` to 0, instead of just failing with an explained error.

- **New `frontend/dagster_wake.py`**: `wake_orchestration()` patches the `autoscaling.keda.sh/paused-replicas` annotation on both `ScaledObject`s to `"1"` (the only mechanism that actually sticks against KEDA's HPA — confirmed by this project's own pre-existing `_wake-orchestration` recipe), then polls both Deployments for `ready_replicas >= 1` up to a 120s timeout, via the Python `kubernetes` client (first direct use of that library in this repo's application code — previously only a transitive `dagster-k8s` dependency).
- **New cross-namespace RBAC**: a dedicated `frontend` ServiceAccount (`frontend/k8s/serviceaccount.yaml`, replacing the shared `default` SA frontend ran as before) plus a `resourceNames`-scoped Role/RoleBinding in the `orchestration` namespace (`orchestration/k8s/rbac-frontend-wake.yaml`) granting `get`/`patch` on exactly the two `ScaledObject`s and `get` on the two Deployments — tighter-scoped than the pre-existing `dagster-run-launcher` Role, deliberately, since this grants a different, lower-trust namespace's identity access.
- **Sleep is deliberately not owned by Streamlit** — a real correctness bug found during planning, not just at the margins: `master_pipeline`'s own run pod calls back into the webserver's GraphQL API for its entire duration (`dagster_launch.py`, up to 1800s), not just at initial submission, so unpausing right after Streamlit's own submit call returns would break the run mid-flight once KEDA's `cooldownPeriod` elapsed. Sleep is owned instead by a new Dagster run-status sensor (`orchestration/dagster_data_platform/dagster_data_platform/wake_sleep_sensor.py`, `ALL_WAKE_SLEEP_SENSORS`, wired into `definitions.py`) — `dagster-daemon` is already always-up and never a KEDA target. Fires on `master_pipeline` reaching `SUCCESS`/`FAILURE`/`CANCELED` (`monitored_jobs=[master_pipeline]`, not every job — a child job like `extraction_*_job` reaching its own terminal status must not trigger this), and only unpauses if Dagster's own run storage (`RunsFilter(job_name="master_pipeline", statuses=<non-terminal>)`) shows no other `master_pipeline` run still in flight — needs no locking for the multi-user case, and survives daemon-pod restarts since it's gated on real persisted state, not an in-process counter. The daemon's existing `dagster-run-launcher` Role (`orchestration/k8s/rbac.yaml`) was extended with the matching `get`/`patch` grant on the same two `ScaledObject`s, same namespace, no new RoleBinding needed.
- **A second real race found live during verification, not anticipated in the original design**: `RunsFilter` alone wasn't sufficient. A sleep-sensor tick processing an *older*, unrelated `master_pipeline` run's terminal-status event can fire in the exact window after a fresh wake but before that new trigger's own run has actually been created for `RunsFilter` to see — from the sensor's point of view, "no other runs in flight" looks true even though a wake is actively in progress. Confirmed live twice, reproducibly: a freshly-created `dagster-webserver` pod was killed ~17s after starting, well before its own readiness probe could ever pass. **Also confirmed to affect this project's own pre-existing `verify-pipeline` tooling**, not just the new Streamlit feature — the identical race hit the gap between `verify-pipeline`'s own sequential `master_pipeline` invocations, killing the webserver mid-poll with a raw `ConnectionResetError`. Fixed by stamping a `data-platform.internal/last-woken-at` timestamp annotation alongside the pause annotation on *every* wake path — both `dagster_wake.py` (Python) and `orchestration/module.just`'s `_wake-orchestration` recipe (shell, `kubectl annotate`) now set it — and having the sleep sensor (`_recently_woken()`) honor a 60s grace period keyed off it before unpausing, regardless of what `RunsFilter` alone says. Self-healing by construction: if a wake is never followed by a real submission, the grace period simply expires and a later, unrelated `master_pipeline` completion event re-evaluates normally: no permanent "stuck awake" leak.
- **New unit test coverage, first k8s-mocking pattern in this repo's tests** (every other test file is a live-cluster/live-Postgres integration test by design): `frontend/tests/test_dagster_wake.py` (5 tests, mocked `kubernetes` client) and `orchestration/dagster_data_platform/tests/test_wake_sleep_sensor.py` (13 tests) — the latter exercises `_sleep_if_no_other_runs_in_flight` directly against a real ephemeral `DagsterInstance` (`dagster._core.test_utils.instance_for_test`/`create_run_for_test`) rather than via `build_run_status_sensor_context`, since that helper needs a genuine `DagsterEvent` from an actually-executed job and the function under test only ever reads `context.instance`/`context.log`. Whether the three sensors are correctly scoped to `master_pipeline` only is checked declaratively against `RunStatusSensorDefinition._monitored_jobs` (no public accessor exists) rather than by invocation, since `monitored_jobs` filtering happens in Dagster's own sensor-evaluation layer, not inside the decorated function.
- **Live verification, full `just smoketest` (genuine nuke, rebuild) green** after both the initial build and the grace-period fix: `verify-pipeline` 5/5 feed-run + 3/3 domain model-run, `verify-schedule`/`verify-sensor` both succeeded, 49/49 dbt tests, all pytest suites (11 `raw_to_clean` + 27 `orchestration` + 15 `frontend` + 6 `tests/integration`), streaming tests all green. The dedicated cold-start test (`frontend/tests/test_trigger_pipeline_page.py::test_trigger_button_wakes_orchestration_from_a_cold_start`, using a new `force_cold_start` fixture that deterministically forces both `ScaledObject`s to 0 rather than waiting on real cron windows) confirmed a genuine cold trigger — Kubernetes API wake, pod rollout, GraphQL submission, real pipeline completion — end to end, twice: once reproducing the race (pre-fix), once passing cleanly in 44s (post-fix).
- **Also found and fixed in passing, unrelated to this feature**: a repo-wide set of stale iCloud "conflicted copy" duplicate files (`* 2.sql`/`* 2.yml`, dated one day prior, owner-only permissions — the same underlying iCloud-sync-on-`~/Documents` cause as this project's known `.pth`/`UF_HIDDEN` bug, but a distinct manifestation: whole duplicate files, not a hidden-flag race) were blocking `dbt` compilation for the `metadata` and `sales` domains (`schema.yml` describing the same resource twice). Deleted outright — all were stale, superseded by the same day's freshly-regenerated codegen output.

## Catalog-based Postgres schema discovery + schema_registry fixes (2026-07-21) — built and live-verified

Closes the Backlog.md item "Catalog-based schema discovery for Postgres — not built." `PostgresConnector.discover_schema()` now queries `pg_attribute`/`pg_type` directly (via `%s::regclass`, same resolution style as the pre-existing `discover_primary_key()`) for the single-real-table case, falling back to the shared Polars sample-inference default only for a custom multi-table `query`. Bundled with four related fixes surfaced while scoping this work, all discussed and confirmed before implementation:

- **`nullable` is real now, not hardcoded `true`**: the catalog path reads `NOT a.attnotnull` directly; the sample-inference path (`connectors/inference.py`) computes it from observed nulls in the fetched data (`df[name].null_count() > 0`).
- **`description` removed from `column_definitions` entirely** — nothing in the platform ever read it for `schema_registry` (only `lakehouse_models` has a real use for descriptions).
- **A column with no data this run no longer silently defaults to `"string"` if it already has a recorded type**: `infer_column_definitions()` now returns `data_type=None` as a sentinel for an all-null column, resolved by `schema_registry_sync.compute_schema_sync()` — a brand-new column still defaults to `"string"`, but an existing column keeps its last real recorded type instead of being overwritten. `nullable` merges one-directionally (`false -> true` only) for the same reason: a run without an observed null doesn't disprove a nullability already on record.
- **`PostgresConnector.fetch()` now passes `infer_schema_length=None`** to its `pl.DataFrame(...)` construction — every row was already fetched into memory, so there's no cost to having Polars infer dtypes from all of them instead of its own default first-100-rows window (mirrors what `csv.py` already did correctly for `pl.read_csv`).
- **New `data_feed.schema_discovery_enabled`/`streaming_source.schema_discovery_enabled` columns** (both default `true`): when off, extraction/discovery skips `discover_schema()`/`discover_primary_key()` entirely and reuses `schema_registry`'s current row as-is — meant to be turned off once a feed or stream's schema is deemed stable. Wired into both frontend CRUD pages, the tabular/JSON codegen templates (`scripts/generate_dagster_pipeline.py`), the `customers`/`sales` stub assets, and `streaming/testing/run.py`'s discovery harness.
- Postgres's catalog-based discovery also runs regardless of whether this run's fetched DataFrame is empty (unlike the sample-based path, which genuinely needs rows) — a Postgres feed's `schema_registry` can now be populated even before its source table ever holds a row.

**Live verification**: `pipeline_generated.py` regenerated against a live metadata Postgres; full `just smoketest` chain confirmed green, run step by step rather than as one combined invocation (`verify-pipeline` → `verify-schedule` → `verify-sensor` → `just test` → `streaming-testing::test`) after an unrelated transient failure on the first combined attempt (see below) — 5/5 feed-run + 3/3 domain model-run rows, 49/49 dbt tests (including `tests/integration/test_ods_primary_key_discovery.py`'s 4 live tests directly exercising `discover_primary_key()`), 27/27 `orchestration` unit tests, 15/15 `frontend` unit tests, 6/6 `tests/integration`, streaming tests all passing.

## Completed `dagster-run-*` pods left running well past when they were needed (2026-07-22) — fixed, live verification pending

Closes the `Backlog.md` item of the same name. Dozens of `dagster-run-*` pods were observed sitting `Completed`, 22-25 hours old, still present in the cluster. First assumption (written into `Backlog.md`/`Roadmap.md` before checking) was that `K8sRunLauncher`'s Job spec had no `ttlSecondsAfterFinished` at all — wrong, corrected after actually reading the installed `dagster_k8s` source: `dagster_k8s/job.py`'s own `DEFAULT_JOB_SPEC_CONFIG` already sets `ttl_seconds_after_finished: 86400` (1 day) on every Job it constructs, unconditionally, via `construct_dagster_k8s_job`'s `merge_dicts(DEFAULT_JOB_SPEC_CONFIG, user_defined_k8s_config.job_spec_config)`. Kubernetes' own `TTLAfterFinished` controller was already going to clean those pods up — they simply hadn't crossed the 24h mark yet when observed. The real ask, once clarified: don't wait a day, delete a run's Job/Pod the moment it finishes.

**Fix**: `run_launcher.config.run_k8s_config.job_spec_config.ttl_seconds_after_finished: 0` added to both `orchestration/dagster_home/dagster.yaml` and `dagster-incluster.yaml` — `run_k8s_config` (confirmed against `DagsterK8sJobConfig.config_type_run_launcher()`'s own config schema, not the per-job `dagster-k8s/config` tag mechanism) is the launcher-level default applied to every job it launches. `0` is Kubernetes' documented value for "eligible for cleanup immediately," not a disabled/no-op sentinel. Trade-off, stated plainly in both files' comments: `kubectl logs` on a completed run's pod stops being possible almost immediately after it finishes — Dagster's own Postgres-backed run/event storage (what `data_processing_runs` and the webserver UI both already read) is unaffected, only the raw pod log stream goes away.

**Live-verified**: updated both ConfigMaps (`dagster-instance`/`dagster-instance-incluster`) and restarted `dagster-daemon` to pick up the new instance config, then triggered a real run via `verify-sensor`. Watched the launched pod's full lifecycle live (`kubectl get pods -w`): reached `Completed` at 20:41:43, confirmed gone (`NotFound`) by 20:42:00 — under a minute, Kubernetes' `TTLAfterFinished` controller cleaning it up on its very next reconcile sweep, versus the old default 24h wait. Pre-existing completed pods from before the config change were untouched (the new TTL only applies going forward, to jobs launched after it took effect), as expected.
